## 🗂️ Sample A: 违约与赔偿剥离 (违约金不平权陷阱)

**📝 待审片段**：
> “若乙方未按期交付，每日按照万分之五支付违约金。若甲方未按期付款，每日按照千分之一支付违约金。任何一方违约金最高不超过合同总额的 20%。”

- **🟢 普通 Prompt 判定**：指出违约金比例不对等，建议改为双方一致，指出 20% 封顶合法但不一定合理。
- **⚡️ v1.2 单兵版**：除了指出不对等，还测算了在 500 万合同下，一旦纠纷被拖入司法程序，20% 封顶会导致甲方最多只能获赔 100 万。即便损失达 300 万，也会因违约金天花板吃闷亏。提供了包括“兜底实际损失”的 Plan B 修改方案。
- **🔥 v2.0 架构版**：
  - **合规雷达 (Agent-1)**：触发违约金最高管辖界限警告，依据民法典结合损失填平原则拦截。
  - **情景沙盘 (Agent-2)**：计算出甲方违约金费率多扛了一半的风险差额。在 Step 6B 模拟出谈判价差。
  - **对抗攻击 (Agent-3)**：模拟乙方律师反击（“你们是甲方，拨款审批慢，我们被拖死怎么办？”），给出**平衡让步树**——可以同意甲方千分之一的万分比，但必须加入“宽限期条款”（即收到合规发票与验收报告后起算的30天免责流转期），实现防御反制。

## 🗂️ Sample B: 业务流程沉默 (验收期限无底洞)

**📝 待审片段**：
> “乙方完成软件开发并部署后，甲方应当进行系统验收。系统验收合格后的 5 个工作日内，甲方需支付尾款。”

- **🟢 普通 Prompt 判定**：提醒缺少具体的验收标准，也只字未提多长时间内要验收完。
- **⚡️ v1.2 单兵版**：严厉警告“默示允诺陷阱”（即无限期拖延付款）。直接起草包含“提交后 N 个工作日内不反馈即视为验收合格”的单方 protection 防御网（Plan B）。
- **🔥 v2.0 架构版**：
  - **底层自洽检查 (Agent-1)**：抓出“验收合格”在司法实务中是事实状态，没有标准就等于扯皮。
  - **对抗漏洞修复 (Agent-3)**：切换成己方防卫视角，压测 Plan B，确保在“视为验收合格”的触发条件中，排除了“因甲方未及时提供测试服务器或 API 导致延误”这类的不可控推诿责任。
  - **生命周期挂载 (Agent-4)**：提取关键节点自动构建 **履约日历 (.ics)**：`[部署之日+ N 日] 触发预警：由法务与项目经理联备函主张验收合格，生成尾款催款邮件底稿`。解决系统离线履约脱节痛点。

## 🗂️ Sample C: 核心 IP 模糊界限 (所有权与使用权混同)

**📝 待审片段**：
> “为本协议项目定制开发产生的所有知识产权及软件源代码，由甲乙双方共同所有。”

- **🟢 普通 Prompt 判定**：指出“共同所有”在后续授权和对外融资时可能会有麻烦，必须说清楚。
- **⚡️ v1.2 单兵版**：援引《著作权法》发出高危 🚨 警告，共同拥有的知识产权在行使商业变现权时需协商一致，这意味着甲方花钱做的系统，一旦乙方不点头，甲方连给三方授权都不行。给出了甲方独占的 Plan B 修改方案。
- **🔥 v2.0 架构版**：
  - **三方利益切分 (Agent-1 & 2)**：评估到 800 万项目的核心资产价值流失，将 EL 设定为“全损”（即丧失未来并购与控制基石），列为最高优先级红线。
  - **BATNA 谈判树策略 (Agent-3)**：
    - 🔴 **绝对红线**：底层源码与业务逻辑专属权利，必须 **100% 归属甲方唯一所有**。
    - 🟢 **可让步回旋**：若是遇到乙方拒签或抬高价格，可附条件让步——允许乙方在“非竞业且剔除所有甲方业务标识、商业机密”的前提下，获得组件级内部复用的普通许可。既守住了红线，又达成商业变现握手。
